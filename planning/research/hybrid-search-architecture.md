# Research: Hybrid Search Architecture (Direct vs. API)

**Status**: Proposal
**Created**: 2026-01-03
**Context**: The user envisions a split query path:
1.  **Simple Queries** (`list`, `filter`): Handled directly by the Go CLI (reading files/index).
2.  **Smart Queries** (`correlated`, `search`): Handled by the Python "Brain" API.

## The Trade-off

### Option A: API Gateway (All traffic via Python)
- **Flow**: `CLI -> HTTP -> Python API -> DB/Index`.
- **Pros**: Consistency. The API is the single source of truth. Logic lives in one place.
- **Cons**: Latency. Even localhost HTTP adds overhead. Requires Docker to be running for *any* command.

### Option B: Hybrid (Direct Read + API) ⭐ **Recommended**
- **Flow**:
    - **Simple**: `CLI -> Reads JSON Index`. (Blazing fast, works if Docker is down).
    - **Smart**: `CLI -> HTTP -> Python API`. (Requires Docker).
- **Mechanism**:
    - The Python service (Watcher) updates `planning-index.json`.
    - The Go CLI treats `planning-index.json` as a read-only database.

## Why this works for TaskFlow
Since `planning-index.json` is a simple flat file generated by the watcher, the Go CLI can easily parse it into structs (`[]Task`).

### Scenario 1: `taskflow list --tag frontend`
1.  Go CLI reads `planning-index.json`.
2.  Filters in memory.
3.  Renders Bubble Tea list.
4.  **Time**: < 10ms. **Docker**: Not required.

### Scenario 2: `taskflow related --semantic "auth logic"`
1.  Go CLI detects semantic flag.
2.  Checks if API is up (`localhost:8000/health`).
3.  Sends `POST /search { "query": "auth logic" }`.
4.  Python API embeds query, searches pgvector, returns IDs.
5.  Go CLI renders results.
6.  **Time**: ~100ms. **Docker**: Required.

## Design Decision
We will implement **Option B**.
- The Go CLI will have a `taskloader` package that reads the JSON index directly.
- It will also have an `apiclient` package for talking to the Brain.
- This ensures the tool feels "instant" for 90% of daily use cases.
